# prd-writer

通过问答引导生成结构化的 PRD 文档。

## 输入

用户描述功能需求，可以是简单一句话，也可以是详细的想法。

## 输出

完整的 PRD 文档（Markdown 格式），可直接用于团队评审。

## 执行步骤

1. 询问基本信息（如果用户没有提供）：
   - 功能名称
   - 一句话描述这个功能解决什么问题
   - 目标用户是谁
2. 根据描述自动生成 PRD 草稿，包含以下章节：

   ```markdown
   # [功能名称] — 产品需求文档

   ## 1. 背景与目标
   ### 1.1 背景
   [为什么要做这个功能，用户痛点是什么]
   ### 1.2 目标
   [核心指标，如何衡量成功]
   ### 1.3 非目标
   [明确不包含什么，避免范围蔓延]

   ## 2. 用户场景
   [2-3 个典型使用场景，用"作为...我想要...以便..."格式]

   ## 3. 功能描述
   ### 3.1 核心流程
   [主流程描述，可配合流程图]
   ### 3.2 功能清单
   | 功能点 | 优先级 | 描述 |
   |--------|--------|------|
   ### 3.3 交互说明
   [关键页面/状态的交互逻辑]

   ## 4. 数据需求
   [需要哪些数据字段、存储、接口]

   ## 5. 验收标准
   [每个功能点的通过条件，可直接转化为测试用例]

   ## 6. 排期与优先级
   | 阶段 | 内容 | 预估工期 |
   |------|------|----------|

   ## 7. 风险与依赖
   [潜在风险、技术依赖、外部依赖]

   ## 8. 附录
   [竞品参考、设计稿链接、技术方案链接]
   ```

3. 展示给用户，逐章节确认或修改
4. 用户确认后保存为 Markdown 文件

## PRD 模板

固定的 PRD 结构模板见 `assets/prd-template.md`，生成时以该模板为骨架填充内容，
确保每次输出的 PRD 格式一致、章节完整。

## 注意事项

- 验收标准要具体可测，不要写"用户体验好"这种模糊标准
- 非目标和目标同样重要，帮用户想清楚不做什么
- 如果用户只给了模糊的想法，先帮他梳理清楚再写文档
- 保持简洁，PRD 不是小说，团队能快速理解最重要
