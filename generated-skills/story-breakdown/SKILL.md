# story-breakdown

把需求拆解为可执行的用户故事和开发任务。

## 输入

用户提供需求描述、PRD 文档、或功能概述。

## 输出

结构化的 Story 列表，包含描述、验收标准、优先级和工作量评估。

## 执行步骤

1. 理解需求全貌：
   - 如果有 PRD 文件，先读取分析
   - 如果是口头描述，先确认理解正确
2. 按用户角色和功能模块拆解 Story：

   ```markdown
   ## Epic: [功能名称]

   ### Story 1: [标题]
   **作为** [角色]，**我想要** [功能]，**以便** [价值]

   **验收标准：**
   - [ ] [具体可测的标准 1]
   - [ ] [具体可测的标准 2]

   **优先级：** P0 / P1 / P2
   **预估工作量：** S / M / L / XL
   **技术备注：** [开发需要注意的点]

   ---

   ### Story 2: [标题]
   ...
   ```

3. 按依赖关系排列执行顺序，标注前置依赖
4. 生成 Sprint 分配建议：
   ```
   Sprint 1（基础能力）：Story 1, 2, 3
   Sprint 2（核心体验）：Story 4, 5
   Sprint 3（增强优化）：Story 6, 7
   ```
5. 展示给用户确认，可增删改任何 Story

## 注意事项

- 每个 Story 应该在 1-3 天内可完成，太大要继续拆
- 验收标准要具体到可以直接写测试用例
- 工作量评估用 T-shirt Size（S/M/L/XL），比精确估时更实际
- 前后端分离的项目，标注每个 Story 涉及前端还是后端还是全栈
- 不要遗漏非功能性需求（性能、安全、监控）
