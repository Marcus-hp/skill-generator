# 架构模式与选型参考

## 服务架构决策树

```
你的项目需要什么架构？

团队 ≤ 5 人？
├── 是 → 单体优先（Modular Monolith）
│   └── 流量超过单机极限？
│       ├── 否 → 保持单体，垂直扩容
│       └── 是 → 拆分瓶颈模块为独立服务
└── 否 → 看业务域复杂度
    ├── 域清晰可拆（如：订单、支付、物流） → 微服务
    ├── 事件驱动为主（如：日志、通知） → Serverless / 事件架构
    └── 不确定 → 先 Modular Monolith，后续再拆
```

## 缓存策略选型

| 场景 | 方案 | 说明 |
|------|------|------|
| 单实例热数据 | 进程内缓存（Map / LRU） | 最快，但重启丢失、多实例不共享 |
| 多实例共享 | Redis | 最通用，支持过期、发布订阅 |
| 高频读低频写 | Redis + 本地缓存（二级缓存） | 减少网络往返 |
| 全页面/接口缓存 | CDN + Edge Cache | 静态资源和公开页面 |
| 数据库查询缓存 | 应用层缓存 + TTL | 不推荐依赖数据库自带查询缓存 |

### 缓存常见问题

| 问题 | 描述 | 解决方案 |
|------|------|---------|
| 缓存穿透 | 查询不存在的数据，每次都打到 DB | 布隆过滤器 / 缓存空值（短 TTL） |
| 缓存雪崩 | 大量 key 同时过期 | TTL 加随机偏移 / 热 key 永不过期 |
| 缓存击穿 | 热 key 过期瞬间大量请求打到 DB | 互斥锁（singleflight）/ 热 key 永不过期 |
| 数据不一致 | 缓存和 DB 数据不同步 | Cache Aside 模式：先更新 DB，再删缓存 |

## 消息队列选型

| 维度 | Redis Streams | RabbitMQ | Kafka |
|------|--------------|----------|-------|
| 吞吐量 | 中（万级/s） | 中（万级/s） | 高（百万级/s） |
| 延迟 | 极低（亚毫秒） | 低（毫秒） | 中（毫秒~十毫秒） |
| 消息持久化 | 可选 | 可选 | 默认持久化 |
| 消费模式 | 消费者组 | 灵活路由 | 消费者组 + 分区 |
| 运维复杂度 | 低（已有 Redis） | 中 | 高 |
| 适合场景 | 轻量异步任务、实时通知 | 复杂路由、任务队列 | 日志流、事件溯源、大数据管道 |

**推荐路径：**
```
小项目/已有 Redis → Redis Streams
需要复杂路由/延迟队列 → RabbitMQ
日志/事件流/大吞吐 → Kafka
```

## 数据库选型

| 维度 | PostgreSQL | MySQL | MongoDB | Redis |
|------|-----------|-------|---------|-------|
| 类型 | 关系型 | 关系型 | 文档型 | KV |
| 复杂查询 | 强（JSON、全文搜索、CTE） | 中 | 中（聚合管道） | 弱 |
| 写入性能 | 高 | 高 | 高 | 极高 |
| 事务支持 | 完整 ACID | 完整 ACID | 多文档事务（4.0+） | 有限 |
| 适合场景 | 通用首选 | 已有 MySQL 生态 | Schema 灵活、嵌套文档 | 缓存、会话、排行榜 |

**默认推荐：PostgreSQL**——功能最全面，JSON 支持好，扩展丰富（PostGIS、pg_vector）。

## API 设计模式

| 模式 | 适合场景 | 不适合场景 |
|------|---------|-----------|
| REST | CRUD 为主的资源操作 | 复杂关联查询、实时双向通信 |
| GraphQL | 前端需要灵活查询、多端适配 | 简单 CRUD、文件上传 |
| gRPC | 内部服务间通信、高性能 | 浏览器直连（需 gRPC-Web） |
| WebSocket | 实时双向通信（聊天、协作） | 请求-响应模式 |
| SSE | 服务端单向推送（通知、进度） | 客户端需要发消息给服务端 |

## 常见规模阈值

帮助判断何时需要升级架构：

| 指标 | 小规模 | 中规模 | 大规模 |
|------|--------|--------|--------|
| DAU | < 1万 | 1万~100万 | > 100万 |
| QPS | < 100 | 100~10,000 | > 10,000 |
| 数据量 | < 1GB | 1GB~1TB | > 1TB |
| 团队 | 1~5人 | 5~30人 | > 30人 |
| **架构建议** | 单体 | 模块化单体或少量微服务 | 微服务 + 基础设施平台 |

## 高可用模式

| 模式 | 描述 | 实现方式 |
|------|------|---------|
| 限流 | 防止突发流量压垮服务 | 令牌桶 / 滑动窗口 / Nginx limit_req |
| 熔断 | 下游故障时快速失败 | Circuit Breaker（Hystrix 模式） |
| 重试 | 临时故障自动恢复 | 指数退避 + 最大次数 + 幂等保证 |
| 降级 | 核心功能保可用 | 返回缓存数据 / 简化版本 / 静态兜底页 |
| 隔离 | 故障不扩散 | 线程池隔离 / 舱壁模式 / 独立部署 |
