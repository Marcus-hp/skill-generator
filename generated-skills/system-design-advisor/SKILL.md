# system-design-advisor

在架构决策点提供结构化分析，帮助做出合理的技术选型和系统拆分。

## 输入

用户描述系统需求、业务场景或某个具体的架构疑问。

## 输出

结构化的架构建议，包含方案对比、推荐理由、风险提示和落地步骤。

## 执行步骤

1. 了解背景，提问关键约束（如果用户没有提供）：
   - 预估 QPS / 用户量级
   - 团队规模和技术栈偏好
   - 预算和运维能力
   - 一致性要求（强一致 vs 最终一致）

2. 根据问题类型，从 `references/design-patterns.md` 中匹配对应的架构模式和决策框架

3. 输出结构化方案：

   ```
   ## 架构建议

   ### 问题定义
   [一句话描述要解决的核心问题]

   ### 方案对比
   | 方案 | 优势 | 劣势 | 适合场景 |
   |------|------|------|---------|

   ### 推荐方案
   [推荐哪个，为什么]

   ### 架构图
   [用 ASCII 或 Mermaid 画出关键组件和数据流]

   ### 落地步骤
   1. [第一步做什么]
   2. [第二步做什么]

   ### 风险与注意事项
   - [风险 1]
   - [风险 2]
   ```

4. 和用户讨论确认，根据反馈调整方案

## 决策领域

**服务架构：**
- 单体 vs 微服务 vs Serverless
- 服务间通信：REST vs gRPC vs GraphQL vs 消息队列
- API 网关设计

**数据层：**
- 数据库选型：PostgreSQL vs MySQL vs MongoDB vs DynamoDB
- 缓存策略：本地缓存 vs Redis vs 多级缓存
- 数据同步：CDC、Event Sourcing、CQRS

**中间件：**
- 消息队列选型：Redis Pub/Sub vs RabbitMQ vs Kafka
- 搜索引擎：Elasticsearch vs Meilisearch vs Typesense
- 任务调度：Celery vs BullMQ vs 云原生方案

**可靠性：**
- 限流熔断：令牌桶 vs 滑动窗口
- 重试策略：指数退避 + 死信队列
- 数据备份与灾备

## 参考框架

完整的架构模式参考、选型决策树和常见场景的推荐方案，
详见 `references/design-patterns.md`。

## 注意事项

- 不要过度设计，日活 1000 的应用不需要微服务
- 先问清约束再给方案，不要上来就推荐最复杂的架构
- 给出方案时要说明"什么量级下该升级到下一个方案"
- 架构图尽量画出来，比纯文字更直观
- 技术选型要考虑团队熟悉度，不只看技术先进性
