# auto-retry-loop

Ralph Wiggum 风格的自查自检循环：持续修改代码并运行验证，直到成功或达到重试上限。

## 输入

用户给出任务描述，或指定要修改的文件/功能。常见表述：
- "帮我改到测试通过"
- "这个 bug 修到能跑为止"
- "循环重试直到构建成功"
- "不要停，改到过为止"

## 输出

持续迭代直到验证通过，或达到最大重试次数。最终输出迭代总结。

## 执行步骤

1. **确认任务和成功条件**：
   - 成功条件是什么？（测试通过 / 构建成功 / 无 linter 错误）
   - 验证命令是什么？（`npm test` / `pytest` / `npm run build` 等）
   - 如用户未指定，根据项目类型推断（package.json → npm test；pyproject.toml → pytest）

2. **设置循环参数**：
   - 最大重试次数：默认 5，用户可指定
   - 失败模式检测：连续 3 次犯同样错误 → 停止并报告，避免死循环

3. **进入循环**：
   ```
   第 N 次迭代：
   1. 分析当前问题（测试失败原因 / 构建错误 / linter 报错）
   2. 修改代码
   3. 运行验证命令
   4. 若通过 → 输出「✅ 完成」，结束
   5. 若失败 → 记录错误模式，判断是否与上次相同
      - 若连续 3 次相同 → 停止，报告「可能需人工介入」
      - 否则 → 进入第 N+1 次迭代
   6. 若达到最大次数 → 停止，报告「已达上限，建议人工检查」
   ```

4. **输出总结**：
   ```
   ## 迭代总结
   - 总迭代次数：N
   - 最终状态：✅ 通过 / ⚠️ 已达上限 / ⛔ 检测到重复失败
   - 修改的文件：[列表]
   ```

## 安全阀

| 机制 | 默认值 | 说明 |
|------|--------|------|
| 最大重试次数 | 5 | 防止无限循环 |
| 重复失败检测 | 连续 3 次相同错误 | 避免死循环，提示人工介入 |
| 用户可中断 | 随时 | 用户说"停"即可终止 |

## 适用场景

**适合：**
- 修 bug 直到测试通过
- 补全功能直到构建成功
- 修复 linter 报错直到通过

**不适合：**
- 架构决策、需要讨论的方案
- 模糊需求、无法程序化验证的"完成"
- 紧急生产事故（需人工快速判断）

## 注意事项

- Trae 可能没有 auto-run，每次运行命令需用户确认；在支持 auto-run 的环境下可自动执行
- 迭代会消耗 token，大任务建议用户知晓
- 成功条件必须可程序化验证，不要依赖 AI 的自我判断
- 规则：若你能精确描述"完成"是什么，此技能可迭代直到达成
