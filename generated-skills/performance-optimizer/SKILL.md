# performance-optimizer

诊断性能瓶颈并给出针对性优化方案。

## 输入

用户描述性能问题（如"接口响应慢"、"首屏加载 5 秒"），或提供性能指标截图/日志。

## 输出

结构化的诊断报告和优化方案，按影响程度排序。

## 执行步骤

1. 确定问题类型：
   - 前端性能 → 加载速度、渲染性能、打包体积
   - 后端性能 → 接口响应、CPU/内存、并发能力
   - 数据库性能 → 慢查询、索引、连接池
   - 全链路 → 从用户请求到响应的完整路径

2. 收集诊断信息（引导用户提供或直接查看代码）：
   - 前端：检查 bundle 配置、图片资源、组件渲染逻辑
   - 后端：查看接口逻辑、数据库查询、外部调用
   - 数据库：查看 SQL 和索引设计

3. 参考 `references/optimization-playbook.md` 中的诊断清单和优化模式

4. 输出诊断报告：
   ```
   ## 性能诊断报告

   ### 问题概述
   [一句话总结瓶颈在哪]

   ### 🔴 关键瓶颈（优先处理）
   - 问题：[描述]
   - 原因：[根因分析]
   - 方案：[优化方法]
   - 预期收益：[优化后的指标]

   ### 🟡 可优化项
   - ...

   ### 🟢 已经做得好的
   - ...
   ```

5. 如果用户同意，直接执行优化（加索引、改配置、重构代码）

## 覆盖场景

**前端性能：**
- 首屏加载优化（代码分割、预加载、SSR/SSG）
- 打包体积分析与瘦身（tree-shaking、动态导入、依赖替换）
- 渲染性能（虚拟列表、防抖节流、减少重渲染）
- 图片优化（格式、尺寸、懒加载、CDN）

**后端性能：**
- N+1 查询检测与修复
- 接口响应链路分析
- 内存泄漏定位（闭包、事件监听、缓存未清理）
- 连接池配置优化

**数据库性能：**
- EXPLAIN 分析与索引优化
- 慢查询识别与重写
- 分页优化（游标分页 vs OFFSET）
- 读写分离与分库分表判断

## 参考手册

完整的诊断清单、优化模式和常见性能指标参考值，
详见 `references/optimization-playbook.md`。

## 注意事项

- 先定位瓶颈再优化，不要盲目优化
- 给出"预期收益"，让用户判断值不值得做
- 优化有成本，要告知增加的复杂度（如缓存带来的一致性问题）
- 不要一次改太多，每次优化一个点并验证效果
