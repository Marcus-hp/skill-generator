# security-checker

对代码和配置进行安全审查，识别潜在漏洞并给出修复方案。

## 输入

用户指定要审查的文件/项目，或者描述安全方面的关注点。

## 输出

结构化的安全审查报告，按风险等级排序，附带修复代码。

## 执行步骤

1. 确定审查范围：
   - 全项目审查 → 扫描所有源文件和配置
   - 特定文件 → 读取并深度分析
   - 上线前检查 → 按 `references/security-checklist.md` 逐项检查

2. 对照 OWASP Top 10 和语言特定的安全清单检查

3. 检查依赖安全：
   - 查看 package.json / pyproject.toml 中的依赖
   - 建议运行 `npm audit` / `pip-audit` / `safety check`
   - 检查是否有已知的 CVE 漏洞

4. 输出安全报告：
   ```
   ## 安全审查报告

   ### 🔴 高危（必须立即修复）
   - [文件:行号] 风险类型：[描述]
     ▸ 攻击场景：[怎么被利用]
     ▸ 修复方案：[具体代码]

   ### 🟡 中危（尽快修复）
   - ...

   ### 🔵 低危 / 建议
   - ...

   ### ✅ 安全实践已到位
   - ...
   ```

5. 提供修复代码，用户确认后直接修改

## 检查维度

**OWASP Top 10 对照：**
- A01 访问控制失效：越权访问、IDOR
- A02 加密失败：明文存储密码、弱哈希算法
- A03 注入：SQL 注入、命令注入、XSS
- A04 不安全设计：缺少速率限制、缺少输入验证
- A05 安全配置错误：调试模式上线、默认凭据
- A06 脆弱过期组件：已知漏洞的依赖
- A07 认证失败：弱密码策略、缺少多因素认证
- A08 数据完整性失败：不安全的反序列化
- A09 日志监控不足：缺少审计日志
- A10 SSRF：服务端请求伪造

**代码层面：**
- 硬编码密钥/token/密码
- 用户输入未验证/转义
- 不安全的文件操作（路径穿越）
- 不安全的随机数（Math.random 用于安全场景）
- CORS 配置过于宽松
- HTTP 响应头缺失（CSP、X-Frame-Options）

**配置层面：**
- .env 文件是否被 Git 追踪
- 生产环境是否关闭了 debug 模式
- 数据库连接是否使用 SSL
- API 密钥是否有最小权限

## 安全清单

完整的上线前安全检查清单和常见漏洞修复模式，
详见 `references/security-checklist.md`。

## 注意事项

- 发现高危漏洞时要明确警告用户，不能埋在报告中间
- 给出攻击场景说明，让用户理解风险而不只是看到"有问题"
- 修复方案要给具体代码，不要只说"需要做输入验证"
- 不要产生误报焦虑，低风险问题标注清楚严重等级
- 敏感信息（密钥、密码）只指出位置，不要在报告中复制出来
