# code-review-assistant

对指定文件或 diff 进行自动化代码审查，输出结构化审查意见。

## 输入

用户指定要审查的文件、目录、或者一段代码片段。也可以审查 git diff（最近的改动）。

## 输出

结构化的审查报告，按严重程度分级，附带修复建议。

## 执行步骤

1. 确定审查范围：
   - 用户指定了文件 → 读取该文件
   - 用户说"审查最近的改动" → 读取 `git diff`
   - 用户贴了代码片段 → 直接审查
2. 识别代码语言（TypeScript / Python / 其他）
3. 按以下维度逐一检查：

   **安全性：**
   - SQL 注入、XSS、硬编码密钥/token
   - 不安全的 eval / exec 使用
   - Python: pickle 反序列化风险；TS: dangerouslySetInnerHTML

   **性能：**
   - N+1 查询、循环内 await、不必要的重渲染
   - Python: 大列表的重复遍历；TS: 缺少 useMemo/useCallback

   **类型安全：**
   - TypeScript: any 滥用、缺少类型守卫、as 强转
   - Python: 缺少类型标注、Optional 未处理 None

   **错误处理：**
   - 裸 except/catch、吞掉错误、缺少边界处理

   **代码规范：**
   - 命名不一致、魔法数字、函数过长、重复代码

4. 生成审查报告：
   ```
   ## 代码审查报告

   ### 🚨 严重问题（必须修复）
   - [文件:行号] 问题描述 → 修复建议

   ### ⚠️ 警告（建议修复）
   - [文件:行号] 问题描述 → 修复建议

   ### 💡 建议（可选优化）
   - [文件:行号] 优化建议
   ```
5. 询问用户是否需要自动修复其中的问题

## 检查清单

TypeScript 和 Python 的完整逐项检查清单（安全性、类型安全、性能、错误处理、代码质量）
以及通用架构和可维护性检查项，详见 `references/review-checklist.md`。

审查时应逐项对照该清单执行，确保不遗漏关键检查维度。

## 注意事项

- 不要报告不影响功能的风格偏好（如单引号 vs 双引号），除非项目有明确的 linter 配置
- 先查看项目的 .eslintrc / pyproject.toml / .prettierrc 等配置，尊重项目已有规范
- 对第三方库的用法给出具体文档引用
- 如果代码量很大，按文件分段审查，不要一次性输出过长报告
