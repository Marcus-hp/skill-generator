# release-notes

自动生成版本发布说明，面向用户和内部团队两个视角。

## 输入

以下任意一种：
- 版本号和时间范围
- 指定 git tag 区间
- 直接描述本次发布的内容

## 输出

两份文档：面向用户的发布说明 + 面向内部的发布清单。

## 执行步骤

1. 收集本次发布的变更信息：
   - 读取 git log（如果有版本 tag 则限定范围）
   - 读取相关的 PRD / Story 文档（如果有）
   - 确认版本号
2. 生成面向用户的发布说明：

   ```markdown
   # v[x.y.z] 更新说明

   发布日期：[YYYY-MM-DD]

   ## ✨ 新功能
   - **[功能名称]**：[用用户能理解的语言描述，不要写技术术语]

   ## 🔧 优化改进
   - [改进描述]

   ## 🐛 问题修复
   - 修复了 [用户可感知的问题描述]

   ## 📢 重要说明
   - [如有破坏性变更或需要用户操作的事项]
   ```

3. 生成面向内部的发布清单：

   ```markdown
   # v[x.y.z] 发布清单

   ## 变更概览
   - 涉及 [N] 个 commit，[N] 个文件变更
   - 关联 Story: [列表]

   ## 部署注意事项
   - [ ] 数据库迁移：[有/无]
   - [ ] 配置变更：[有/无]
   - [ ] 环境变量：[新增/修改的环境变量]
   - [ ] 依赖更新：[有/无]
   - [ ] 缓存清理：[是否需要]

   ## 回滚方案
   [如需回滚的操作步骤]

   ## 测试确认
   - [ ] 核心流程验证
   - [ ] 新功能验证
   - [ ] 回归测试
   ```

4. 展示给用户确认和修改

## 注意事项

- 面向用户的说明不要包含技术细节，用"修复了登录偶尔失败的问题"而非"修复了 JWT 过期未刷新导致的 401"
- 面向内部的清单要包含回滚方案
- 如果有破坏性变更（breaking change），必须醒目标注
- 按功能重要性排序，不要按 commit 时间排序
